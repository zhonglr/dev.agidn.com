# 文档贡献规范

> 状态：Active
> 最后更新：2026-07-23
> 适用范围：根目录 `README.md`、`docs/**/*.md` 以及代码目录中的局部 `README.md`

本文定义文档存什么、放哪里、何时创建或更新，以及哪些内容不得混写。它是文档维护的规范性事实来源。

## 1. 目录与职责

| 目录 | 内容 | 不应包含 |
| --- | --- | --- |
| `docs/product/` | 稳定的产品边界、目标用户和验收定义 | 当前任务清单、单次 UAT 结果 |
| `docs/architecture/` | 当前有效的系统设计、模块边界和迁移方案 | 每次开发流水账 |
| `docs/adr/` | 已接受或废弃的长期技术决策 | 尚未决定的临时想法 |
| `docs/api/` | 当前有效的外部协议、格式和示例 | 与实现不一致的未来接口 |
| `docs/project/` | 当前状态、路线图和 Cycle 记录 | 具体 Issue 的完整复现步骤 |
| `docs/quality/README.md` | 质量记录入口、选择规则，以及 UAT/Verification 索引 | 单个问题的完整历史 |
| `docs/quality/issues.md` | 跨轮次问题的状态、根因和关闭证据 | 整体路线图、一次性验收过程 |
| `docs/quality/uat/` | 某次人工验收的环境、步骤和原始结论 | 会随实现变化的当前状态 |
| `docs/quality/verification/` | 兼容性 Spike、迁移 Gate 和专项验证报告 | 产品 UAT 或长期 Issue 状态 |
| `docs/contributing/` | 对贡献者有约束力的工程和文档规范 | 产品规划 |
| `docs/templates/` | 新记录的起始结构和必填字段 | 已发生事件 |
| `docs/archive/` | 已失效但值得追溯的历史基线 | 当前指导意见 |

代码子目录中的 `README.md` 只解释该目录的局部边界、扩展方式和注意事项，并链接到 `docs/` 中的权威设计；不得复制整篇架构文档。

## 2. 文件和标识命名

- 目录和普通 Markdown 文件使用小写 kebab-case，例如 `studio-workbench.md`。
- `README.md` 是唯一允许大写的普通文件名，只用于目录入口；不能用 `index.md` 与它竞争。
- ADR 使用 `NNNN-short-title.md`，编号四位、只增不复用，例如 `0005-document-lifecycle.md`。
- Cycle 默认使用 `cNN.md`；只有存在本轮专属附件时才使用 `cNN/README.md` 和 `cNN/artifacts/`。超过 99 轮时自然扩展为 `cNNN`，既有 ID 不改名。
- Issue 的稳定 ID 使用 `<DOMAIN>-NNN`，例如 `STUDIO-018`。当前规模集中维护在 `quality/issues.md`；需要独立文件时命名为 `quality/issues/studio-018-short-title.md`，一个文件只描述一个长期 Issue。
- UAT 使用 `<YYYY-MM-DD>-<domain>-round-NN.md`，例如 `quality/uat/2026-07-23-studio-round-03.md`。
- Verification 使用 `<YYYY-MM-DD>-<domain>-<gate-or-topic>.md`，例如 `quality/verification/2026-07-23-studio-ui-gate-01.md`。
- 文件名不使用 `final`、`new`、`latest`、`v2` 之类无法解释生命周期的后缀；版本、日期、轮次或稳定 ID 必须显式。稳定台账类文件（如 `quality/issues.md`）使用领域名词命名，不受此后缀规则限制。

重命名或移动文档时必须同步更新全部仓库内链接。Git 历史承担旧路径追溯，不保留重复副本或只有跳转作用的空壳文件。

## 3. 通用格式

每份文档必须满足：

1. 第一行是唯一的一级标题；正文从二级标题开始。
2. 使用 UTF-8、LF、尾部换行，不保留行尾空格；需要换行时优先分段而非 Markdown 双空格。
3. 标题描述内容，不写“其他”“最新”“临时记录”等模糊名称。
4. 相对链接必须指向仓库内真实文件；链接文字表达目标内容，不使用裸路径充当句子。
5. 命令默认从仓库根目录可执行；例外必须注明工作目录。
6. 表格只用于结构一致的字段；过程解释使用正文或列表，不把长篇说明塞进单元格。
7. 代码、协议字段、路径和 ID 使用反引号；产品可见名称不因是英文而滥用代码格式。

状态类文档必须在标题后提供“最后更新”；不可变记录提供“日期/状态”；归档文档必须醒目标明 `Archived` 并链接当前替代文档。日期统一使用 `YYYY-MM-DD`，状态值使用模板中规定的枚举。

## 4. 文档类型与生成时机

| 类型 | 创建时机 | 更新时机 | 结束方式 |
| --- | --- | --- | --- |
| 产品设计 | 产品边界、目标用户或验收定义需要跨多个 Cycle 保持稳定时 | 产品范围、用户任务或验收定义改变时 | 被新产品定义取代时归档并链接替代文档 |
| 系统架构 | 已实现或已批准的模块边界、数据流、扩展机制需要成为团队共同约束时 | 实现改变当前有效设计的同一提交 | 被新架构取代时更新正文；重大决策另建 ADR |
| 项目状态 | 项目建立时创建一次 | 可运行能力、测试基线或下一步改变时 | 永久维护，不按轮次复制 |
| 路线图 | 有跨 Cycle 计划时创建一次 | 优先级、范围或里程碑改变时 | 已完成项保留摘要，细节转 Cycle |
| ADR | 长期架构选择被接受时 | 只补事实错误或状态；新决策新建 ADR | Superseded 时链接替代 ADR |
| API | 端点/格式进入实现或公开契约时 | Schema、行为或错误边界改变的同一提交 | 协议废弃后移入历史段或归档 |
| Issue | 已有复现/证据，且需要跨提交跟踪、独立优先级、独立验收或后续复验时 | 状态、根因、修复、验证证据变化时 | 只有验收条件全部满足才 Closed |
| UAT | 已锁定 commit、验收人、环境和用户场景，且结论会影响产品接受判断时 | 验收过程中按事实追加 | 结论固定后不重写；后续复验新建轮次 |
| Verification | 存在明确技术问题、通过阈值、可重复方法和独立留证需要时 | 验证期间按证据追加 | 结论固定后只追加 amendment；后续重测新建报告 |
| Cycle | 一组工作有共同交付目标、start commit、范围/非目标，并需要独立测试或 UAT 时 | Dev/Fix/Test/UAT 各阶段结束时 | 锁定 end commit 与 carryover |
| 贡献规范 | 同一工程或文档规则会重复约束多个贡献者，并且可以审查或自动检查时 | 团队规则或工具约束改变时 | 被替代时直接更新当前规范，必要时用 ADR 解释重大变化 |
| 模板 | 某类记录会重复创建，且必填字段已经稳定时 | 规范字段或生命周期改变时 | 不再创建该类记录时删除或归档 |
| Archive | 当前文档已被取代但有审计价值时 | 仅修复链接和事实错误 | 永久只读并链接替代文档 |

代码完成、测试通过和用户验收是三个不同状态。文档不得把 `Development complete` 写成 `Accepted`，也不得用测试数量代替真实交互验收。

## 5. Issue、UAT 和 Cycle 的边界

- Issue 是跨轮次的可变状态；UAT 是一次验收的不可变证据；Cycle 是一段 Git 时间边界。
- UAT 发现问题时创建或关联 Issue，不在 UAT 文件里长期维护修复状态。
- Cycle 只记录本轮 Issue 快照和链接，不复制 Issue 的全部复现/验收内容。
- 关闭 Cycle 不等于关闭 Issue；Issue carryover 后继续使用原 ID。
- 当前 [Studio Issue 台账](../quality/issues.md) 保留早期集中记录。新 Issue 先进入台账；只有跨多个 Cycle、证据很多、多次 Reopened 或明显降低台账可读性时才拆为独立文件，ID 不变。

### 5.1 不需要创建正式记录的情况

- 当前提交内立即修复、风险很低且已有回归覆盖的小问题：写清 commit，不创建 Issue。
- 普通 `typecheck`、单元测试和生产构建：记录到 Cycle Test Log，不创建 Verification。
- 开发者临时冒烟、没有固定 commit 或验收人的操作：不创建 UAT。
- 没有交付边界的长期想法：进入 Roadmap，不创建 Cycle。
- 尚未确认的架构探索：进入评估笔记或 Verification；只有决策被接受后才创建 ADR。
- 单个功能的实现细节：留在代码和测试；只有形成跨模块共同约束时才创建或扩展 Architecture。
- 一次性文档结构：直接写记录，不为它创建模板；至少预期重复使用并已有稳定字段时才增加 Template。

### 5.2 目录扩展门槛

Quality 默认采用扁平文件名表达领域。只有出现第二个长期维护领域，或同类活跃记录超过约 10 份并显著影响浏览时，才增加领域子目录。普通 Cycle 使用单文件，只有真实附件存在时才升级为目录。禁止为空间“预留”创建空目录、空索引和只有一个子项的导航层。

## 6. 更新与审查清单

生成或修改文档时确认：

- 内容属于这个文档类型，没有复制另一份事实来源。
- “当前、已完成、支持、通过”等结论有代码、测试、提交或验收证据。
- 计划项与已实现项明确分开；未知事项写明未知，不补猜测。
- 协议变更同时更新 Schema、契约测试和 API 文档。
- 架构变化有 ADR 或明确说明为何不需要 ADR。
- 新 Issue/UAT/Cycle 使用模板，并加入相应目录索引。
- 文件路径、标题、日期、状态和内部链接通过 `pnpm docs:check`。
- 只修文档时至少运行 `pnpm docs:check`；涉及实现事实时还要运行与改动相称的类型检查、测试和构建。

## 7. 禁止事项

- 不把聊天记录、未经整理的命令输出或本机绝对路径提交为正式文档。
- 不提交包含密钥、访问令牌、用户数据或内部服务地址的日志/截图。
- 不手工维护可从代码稳定生成的大段清单；必须生成时记录来源命令、工具版本和生成日期。
- 不静默改写已关闭 UAT、Cycle 或 ADR 的历史结论；使用 amendment、后续轮次或新 ADR。
- 不以删除不利结论的方式“整理”历史；失效内容应归档、标注或由新记录取代。
