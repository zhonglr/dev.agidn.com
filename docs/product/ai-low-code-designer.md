# AI 友好的低代码网页设计器

> 技术架构、模块边界和目录组织参见 [系统架构](../architecture/system-overview.md)，具体技术选型参见 [技术决策汇总](../architecture/technical-decisions.md)。当前进度和实施顺序分别见 [项目状态](../project/status.md) 与 [项目路线图](../project/roadmap.md)。

## 1. 产品定位

本产品不是一个单纯的拖拽式网页编辑器，而是一个人类优先、面向机器消费的可视化 UI 规格编译器（Visual UI Spec Compiler）。

早期版本由人类通过可视化画布完成页面设计；系统在设计过程中同步生成一份独立、可验证、可移植的 PageDocument。设计完成后，PageDocument 及其必要引用上下文可以交给任意 AI 编码工具使用，AI 不需要依赖截图猜测页面结构。

> 核心目标：人类通过拖拽表达产品意图，系统将其保存为无歧义、可执行、可验证的 PageDocument，使下游 AI 在不追问、不猜测、不创造新规范的情况下实现正确代码。

画布只是 PageDocument 的一个人类客户端，真正的产品资产是独立的结构化文档。产品允许人类自由表达内容、业务和组合意图，但不允许任何客户端随意创造实现方式。实现方式必须来自受控的布局系统、设计令牌、组件变体和真实代码组件。

## 2. 产品价值

传统低代码编辑器主要解决“无需编码也能搭建页面”的问题，但通常存在以下不足：

- 元素以坐标和零散样式保存，AI 很难理解其真实用途。
- 复制组件后样式相互独立，难以长期维护。
- 不同页面不断产生相似但不一致的样式。
- AI 生成内容时容易创造新的颜色、间距和组件形式。
- 页面结构只能被渲染，不能被可靠地解释和复用。

本产品需要把设计结果转化为结构化资产，包括：

- 带有业务和视觉语义的页面结构。
- 可追踪、可引用的设计令牌。
- 支持变体和受控属性覆盖的组件样式。
- 可以跨页面复用的页面模式。
- 明确的响应式规则和布局约束。

## 3. 核心设计原则

### 3.1 语义优先

系统保存的不是单纯的坐标关系，而是元素在页面中的角色和结构关系。

不推荐：

```text
按钮位于 x=820、y=460
```

推荐：

```text
这是产品首屏中的主要操作按钮，位于标题和描述下方，使用品牌主按钮样式。
```

### 3.2 约束优先

页面采用 Stack、Row、Grid、Container 等约束式布局。普通内容节点不提供自由定位；需要叠加的特殊场景只能通过受控 Overlay 和锚点规则表达。

### 3.3 引用优先

组件应引用设计令牌和组件变体，而不是在每个实例上保存一份独立样式。实例只能覆盖组件注册表明确允许的属性；覆盖必须显式记录并接受系统检查。

### 3.4 复用优先

系统不仅复用基础组件，还要支持复用由多个组件组成的页面模式，例如 Hero、价格区、功能列表和登录表单。

### 3.5 所有修改必须可解释

人类操作必须形成可追踪的 Command、Patch 和文档版本。未来 AI 通过 MCP 修改页面时也必须走相同链路，并展示修改内容、原因和影响范围，支持验证、审查和撤销。

### 3.6 非法状态不可表达

样式和布局约束不能只依赖文档、提醒或事后检查，而要落实在页面 Schema 和编辑器能力中。数据模型本身不应支持任意 CSS、无语义坐标、未知组件属性或未登记的设计值。

所有客户端都必须经过同一套类型系统与规则引擎。Studio、后端、未来 MCP 和 AI 都不能绕过规则直接写入页面。

### 3.7 分层事实来源

PageDocument 是页面组合、语义、布局、状态和交互的唯一事实来源；组件注册表是组件能力的事实来源；Token Registry 是设计值的事实来源；Policy Registry 是布局和约束的事实来源。

编辑器中的组件来自真实代码组件注册表，而不是另外维护一套设计组件。PageDocument 只保存引用和页面级配置，不复制组件实现、Token 值或 Policy。各事实来源相互引用但不能互相隐藏覆盖。

## 4. 核心功能

### 4.1 语义化布局

用户拖入画布的对象包括：

- 页面（Page）
- 区块（Section）
- 容器（Container）
- 横向布局（Row）
- 纵向布局（Stack）
- 网格（Grid）
- 导航（Navigation）
- 产品首屏（Hero）
- 功能列表（Feature List）
- 商品列表（Product Grid）
- 表单（Form）

每个节点都可以包含以下信息：

- 节点类型。
- 业务或视觉角色。
- 父子结构。
- 布局规则。
- 样式引用。
- 内容数据。
- 响应式行为。
- 可编辑约束。

### 4.2 设计令牌

系统集中管理以下基础设计规范：

- 品牌色、辅助色、文本色和状态色。
- 字体、字号、字重和行高。
- 间距等级。
- 圆角等级。
- 边框和阴影。
- 页面容器宽度。
- 响应式断点。

系统应检测重复的局部样式，并建议将其提升为设计令牌。例如：

> 检测到 6 个组件使用相同的 16px 圆角，是否保存为 `radius.card`？

设计令牌示例：

```json
{
  "colors": {
    "brand": "#635BFF",
    "textPrimary": "#161616"
  },
  "spacing": {
    "sm": "8px",
    "md": "16px",
    "lg": "32px"
  },
  "radius": {
    "card": "16px"
  }
}
```

### 4.3 样式引用与组件变体

组件实例通过引用获得样式：

```text
按钮实例 -> Button / Primary / Large
```

系统需要支持：

- 创建和编辑组件样式。
- 创建尺寸、状态和用途变体。
- 修改样式后同步更新全部引用实例。
- 在组件注册表允许的范围内覆盖实例属性。
- 查看某个样式被哪些页面使用。
- 将局部样式提升为全局样式。
- 检测脱离设计系统的组件实例。

### 4.4 页面模式

页面模式是比基础组件更高层级的可复用结构，例如：

- 顶部导航。
- 左右双栏 Hero。
- 功能介绍区。
- 价格卡片组。
- 用户评价区。
- 登录或注册表单。
- 后台筛选列表。
- 页脚。

每个模式包含：

- 布局结构。
- 组件组合。
- 样式和变体引用。
- 可编辑内容槽位。
- 使用场景和语义说明。
- 桌面端与移动端规则。

人类可以通过拖拽已有模式快速组合页面；这些模式同时为下游 AI 提供稳定的复用单位。

### 4.5 Schema 理解与完整度检查

系统需要向用户展示当前 PageDocument 对页面或选中区域的结构化描述，例如：

```text
区域角色：产品首屏
布局方式：左右双栏
视觉重点：左侧标题与主要操作按钮
内容层级：标题 > 描述 > 操作按钮
使用模式：Hero / Split
样式来源：Marketing Theme
响应式行为：移动端改为纵向排列
```

用户可以纠正这些语义。修正后的信息直接成为 PageDocument 的一部分，而不是保存在编辑器临时状态中。

### 4.6 未来 MCP 操作接口

MVP 不提供 AI 生成设计的入口。未来 AI 或其他自动化客户端通过 MCP 读取文档并提交结构化 Command，例如：

- 把选中区域改成左右两栏。
- 根据当前设计系统新增一个价格方案。
- 让所有卡片使用统一的圆角和阴影。
- 找出没有复用全局样式的组件。
- 把选中区域保存为可复用模式。
- 生成移动端布局，但不改变桌面端布局。
- 参考首页样式创建联系我们页面。
- 使用现有导航、Hero 和功能卡片组合产品介绍页。

MCP 客户端提交后必须得到结构化验证和变更清单，例如：

- 新增了哪些节点。
- 删除或移动了哪些节点。
- 修改了哪些布局规则。
- 引用了哪些设计令牌或组件变体。
- 哪些修改会影响其他页面。

### 4.7 真实代码组件注册表

项目组件进入编辑器之前必须完成注册。注册信息至少包含：

- 包名、导出名称和源码位置。
- 可用 Props 及其类型和默认值。
- 可用 Slots 及允许放入的内容。
- 组件变体和合法组合。
- Loading、Empty、Error、Disabled 等状态。
- 无障碍要求。
- 组件使用规则和禁止事项。

示例：

```json
{
  "component": "@app/ui/Button",
  "export": "Button",
  "props": {
    "variant": ["primary", "secondary", "danger"],
    "size": ["sm", "md", "lg"],
    "disabled": "boolean"
  },
  "slots": ["label", "leadingIcon"],
  "rules": [
    "每个页面只能有一个主要操作区域",
    "图标按钮必须提供 aria-label"
  ]
}
```

用户拖入组件后，编辑器、编译器和下游 AI 都能通过同一注册信息获得正确的 import、Props、Slots 和约束，不再根据图形外观猜测实现。

### 4.8 封闭式样式系统

组件不开放完整 CSS，只开放组件注册信息中明确声明的样式属性。例如：

```json
{
  "component": "Card",
  "allowedStyleProps": {
    "padding": ["none", "sm", "md", "lg"],
    "radius": ["none", "card"],
    "surface": ["default", "muted", "brand"],
    "shadow": ["none", "sm", "md"]
  }
}
```

每个样式只能来自以下三个来源：

1. Design Token：颜色、间距、圆角、字号和阴影等基础值。
2. Component Variant：例如 `Button.primary.large`、`Card.featured`。
3. Layout Policy：容器宽度、区块间距、网格和响应式规则。

系统不提供“任意填写一个值”作为第四种来源。以下写法不允许进入页面模型：

```tsx
<Card style={{
  padding: "17px",
  borderRadius: "13px",
  background: "#F8F9FC"
}} />
```

当现有设计系统不能满足需求时，任何客户端都必须发起显式变更：

```text
说明使用场景
→ 查找现有 Token 或变体
→ 检测重复定义
→ 申请新增全局 Token 或组件变体
→ 确认后进入设计系统
```

局部例外必须记录理由、影响范围和责任来源，不能悄悄成为不可追踪的样式覆盖。

### 4.9 封闭式布局系统

第一版只允许使用有限的布局节点：

```text
Page
└── Section
    └── Container
        ├── Stack
        ├── Row
        ├── Grid
        └── Overlay
```

所有内容组件必须处于布局节点中。用户拖动组件时，系统修改的是父容器、子节点顺序、布局类型、对齐方式、Token 化间距和响应式规则，而不是 `x`、`y`、`top` 或 `left`。

布局示例：

```json
{
  "type": "grid",
  "columns": {
    "desktop": 3,
    "tablet": 2,
    "mobile": 1
  },
  "gapToken": "spacing.lg"
}
```

系统内置的布局规则包括：

1. 页面主要内容必须处于 `Section > Container` 中。
2. 普通内容节点禁止使用绝对定位。
3. 间距必须引用 Token。
4. Grid 必须声明必要的响应式列数。
5. 组件只能出现在注册表允许的 Slot 中。
6. 布局嵌套深度和嵌套类型必须受到限制。
7. 任何客户端都不能直接写入任意 CSS 布局属性。

### 4.10 受控的绝对定位

绝对定位不完全禁止，但只能用于确实需要叠加的场景：

- Badge 角标。
- Tooltip、Popover、Dropdown 和 Modal。
- 图片上的文字或操作按钮。
- 明确标记为装饰用途的图形。

系统禁止设置任意坐标，只允许通过锚点、边界和 Token 化偏移表达：

```json
{
  "type": "overlay",
  "purpose": "badge",
  "anchor": "top-right",
  "offsetToken": "spacing.sm",
  "boundary": "parent",
  "content": {
    "componentRef": "Badge"
  }
}
```

浮层组件必须引用触发节点并声明碰撞策略：

```json
{
  "componentRef": "Popover",
  "anchorRef": "user-menu-button",
  "placement": "bottom-end",
  "collisionStrategy": "flip"
}
```

装饰元素必须显式标记为 `decoration`，不能承载业务操作，并应默认从无障碍树中隐藏。

### 4.11 结构化写入与规则拦截

任何客户端都不得直接覆盖页面。人类操作和未来 MCP 请求首先转化为结构化 Command，再产生 Patch 并经过验证：

```text
人类拖拽 / MCP 请求
        ↓
生成结构化操作
        ↓
Schema 类型检查
        ↓
布局规则检查
        ↓
Token 与组件检查
        ↓
响应式和无障碍检查
        ↓
允许写入页面
```

非法操作应在写入前拒绝，并返回合法选择：

```json
{
  "accepted": false,
  "reason": "不允许设置任意间距",
  "suggestions": ["spacing.sm", "spacing.md"]
}
```

### 4.12 Schema Context Package

系统不应只向下游 AI 输出截图或一段自然语言 Prompt，而应导出版本化的 Schema Context Package。文件内容与导出规则的权威定义见 [Schema Context Package](../api/context-package.md)。

`document.json` 是页面事实来源，其他文件只用于解析其中的引用。系统只导出当前页面使用或允许使用的组件、Token 和 Policy，避免把整个项目作为噪声交给 AI。

## 5. Studio 工作区

Studio 是专业生产力工具，不使用固定三栏作为产品结构。下图只是内置默认布局，用户可调整每个区域的位置和尺寸，也可移动、停靠、合并、折叠或关闭面板。

```text
┌────────────────────────────────────────────────────────────────────┐
│ 项目 / 页面       设计系统       文档完整度        设备切换  保存 导出 │
├──────────────┬──────────────────────────────────┬──────────────────┤
│ 资源 / 结构  │              设计画布             │ 属性 / Schema    │
│              │                                  │                  │
│ 基础布局      │  Section: Product Hero           │ 当前区域语义      │
│ 页面模式      │  ┌────────────────────────────┐  │ 布局约束          │
│ 项目组件      │  │ 标题          产品图片       │  │ 样式引用          │
│              │  │ 描述                         │  │ 响应式规则        │
│ 页面结构树    │  │ 主按钮                       │  │ 文档检查          │
│ Problems / 历史  │  └────────────────────────────┘  │                  │
├──────────────┴──────────────────────────────────┴──────────────────┤
│ Schema 检查：2 个重复样式 · 1 个未命名区域 · 移动端存在布局风险      │
└────────────────────────────────────────────────────────────────────┘
```

### 5.1 Workbench 布局

- 工作区由数据驱动的 Split、Tab Group 和 Panel 组成，不写死在页面 JSX 中。
- 分隔条可拖动，面板可移动、停靠、合并为标签、折叠和重新打开。
- 用户布局、面板可见性和活动标签会被保存，并可重置为内置预设。
- 内置面板与未来扩展面板通过同一 Panel Registry 接入。

### 5.2 常驻高频操作

- 项目与页面切换。
- 撤销与重做。
- 桌面、平板和移动端切换。
- 当前选中、核心 Props、Variant、布局与响应式配置。
- 文档完整度、预览状态、保存和导出。
- Command Palette、快捷键和稳定状态栏。

### 5.3 中央设计画布

- 拖拽和选择页面节点。
- 展示节点语义、布局边界、辅助线和合法落点。
- 调整约束式布局参数并查看不同响应式断点。
- 支持触控板双指平移、pinch 缩放、指针中心缩放、`100%`、Fit Page 和 Fit Selection。
- 缩放只作用于画布内的 Preview Surface，不能改变 Studio 面板、工具栏和文字尺寸。
- 画布视口可平移和缩放，但页面内容仍使用受控布局，不会变成任意坐标的无限画布。

### 5.4 可编排工具面板

- 资源、页面结构、Inspector、Problems、History 和 Tasks 都是可注册面板。
- 面板应根据当前选中和任务显示上下文，避免把所有配置长期平铺。
- 布局、标签页、分隔条、面板焦点和关键操作必须可通过键盘访问。

### 5.5 独立页面与对话框

- Token、主题、Component Registry、Pattern、Policy、Constraint 和 Data Source 等低频全局配置进入独立页面或大型对话框。
- 导出设置、项目设置、快捷键和完整 Revision 审查按需打开。
- 低频配置不得仅因为能够容纳在侧栏中就长期占用画布宽度。

### 5.6 问题与状态

持续提示：

- 重复但未复用的样式。
- 没有命名或语义不明确的区域。
- 脱离设计系统的组件。
- 响应式布局风险。
- 颜色对比度和基础可访问性问题。

完整 Workbench 布局模型、插件扩展点、Canvas Viewport 输入规则和验收基线见 [Studio Workbench 架构](../architecture/studio-workbench.md)。

## 7. 首版 MVP 范围

首版优先验证一条人类操作闭环：注册真实代码组件，人类通过受限拖拽生成 PageDocument，前后端通过统一 Command 协作，系统验证、保存版本并导出 Schema Context Package。

### 7.1 真实代码组件注册

- 从代码库注册 10～20 个真实组件。
- 提取组件 import、Props、Slots、变体和状态。
- 为组件声明允许的子节点和样式能力。
- 直接使用真实组件渲染画布和预览。

### 7.2 语义化页面 Schema

- 通过拖拽生成结构化节点树。
- 支持 Section、Container、Stack、Row 和 Grid。
- 支持为节点设置名称和语义角色。
- 支持桌面端和移动端布局规则。
- 不在 Schema 中提供普通节点的坐标和任意 CSS 字段。

### 7.3 设计令牌与样式引用

- 管理颜色、文字、间距、圆角和阴影令牌。
- 组件实例可以引用令牌和组件变体。
- 修改全局样式后同步更新实例。
- 检测重复样式和局部覆盖。
- 未登记的设计值不能写入页面。

### 7.4 页面行为与状态

- 支持 Loading、Empty、Error、Success 和 Disabled 状态。
- 支持基础点击、跳转、表单和数据绑定描述。
- 支持为关键元素声明无障碍名称和语义。

### 7.5 前后端与文档独立

- Studio 只通过 Command 操作文档，不直接持久化正式文档。
- Workspace Server 负责验证、版本、事务和持久化。
- `document-schema` 不依赖 Studio、React、后端、Registry、AI 或 MCP。
- Renderer 和导出器只读取文档，不能修改文档。
- 前端可以乐观预览，后端负责最终验证和提交。

### 7.6 Context Package 导出与验证

- 输出 PageDocument、组件注册表、Token、Policy、行为和约束。
- 只导出当前页面实际需要的引用上下文。
- 在写入前完成类型、布局、样式、响应式和无障碍检查。
- 对非法操作给出拒绝原因和合法替代项。

## 8. 首版不优先实现的能力

为避免产品退化成普通搭建器，首版暂不以以下能力为重点：

- 大量业务组件和行业模板。
- 矢量绘图和无限画布。
- 任意绝对定位和坐标编辑。
- 无限制 CSS 属性。
- 与真实代码无关的高保真模拟组件。
- 复杂动画编辑器。
- 海量插件系统。
- 完整后端逻辑编排。
- 数据库和工作流搭建。
- 不受设计系统约束的整页代码生成。

## 9. MVP 验收标准

当用户完成一个简单营销页面后，系统至少能够：

1. 输出包含语义和响应式规则的页面节点树。
2. 画布中的主要组件全部绑定到真实代码组件。
3. 明确指出每个组件使用了哪些设计令牌和变体。
4. 拒绝任意颜色、间距、圆角和 CSS 属性进入页面模型。
5. 普通内容节点无法创建绝对定位或保存 `x/y/top/left`。
6. Overlay 必须声明用途、锚点、边界和 Token 化偏移。
7. Grid 等布局节点具有完整的响应式规则。
8. 检测重复样式、非法局部覆盖和无语义节点。
9. 人类拖拽产生的所有修改都经过 Command、验证、Patch 和版本提交。
10. 前端乐观结果与后端提交结果能够正确确认或回滚。
11. 页面状态、基础行为和无障碍要求能够进入 Schema Context Package。
12. 将现有区块保存为页面模式，并在另一个页面复用。
13. 每次文档修改都具有可检查、可验证和可撤销的变更记录。

首版最重要的产品指标不是画布功能数量，而是：人类能否在不接触任意 CSS 和坐标的情况下完成页面，并稳定导出完整、有效、可供下游 AI 使用的 PageDocument。

## 10. 后续可扩展方向

- 从现有网页或设计稿中提取页面 Schema 和设计令牌。
- 根据组件使用情况自动整理设计系统。
- 设计系统版本管理和升级影响分析。
- 多品牌主题与主题切换。
- 将 Schema 导出为 React、Vue 或标准 HTML/CSS。
- 扩展组件注册协议，支持更多框架和自动提取能力。
- 团队级页面模式库和权限管理。
- MCP Server 和 AI Proposed Commands。
- AI 自动检查一致性、可访问性和响应式质量。
